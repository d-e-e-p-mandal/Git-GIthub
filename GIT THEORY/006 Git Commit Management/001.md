# Commit Structure

# What is a Commit?

A commit is a snapshot of the repository at a specific point in time.

Git commits store:
- File changes
- Metadata
- Project state
- Relationships with previous commits

Each commit becomes part of the repository history.

---

# Purpose of Commits

Commits are used to:
- Save project progress
- Track changes
- Restore previous versions
- Collaborate with teams
- Maintain development history

---

# Commit Structure Components

A Git commit contains:

| Component | Description |
|---|---|
| Commit Hash | Unique identifier |
| Author Name | Creator of commit |
| Author Email | Email of creator |
| Timestamp | Date & time |
| Commit Message | Description of changes |
| Parent Commit | Previous commit reference |
| Tree Object | Snapshot of project files |

---

# Internal Commit Structure

Git stores commits internally as objects.

Commit object contains:
- Tree reference
- Parent reference
- Author metadata
- Commit message

---

# Commit Relationship

Commits are linked together:

```text
Commit A → Commit B → Commit C
```

Each commit points to parent commit.

---

# Commit Snapshot Concept

Git stores:
- Complete snapshot of project

Git does NOT store:
- Only line-by-line changes

Every commit represents full project state.

---

# Commit Metadata

Metadata helps identify:
- Who made changes
- When changes happened
- Why changes were made

---

# Commit Messages

Commit messages explain:
- Features added
- Bugs fixed
- Changes introduced

Good commit messages improve:
- Team understanding
- Debugging
- Project history readability

---

# Example Commit Information

```text
Commit: a1b2c3d4
Author: Deep Mandal
Date: 2026-05-10
Message: Add login authentication
```

---

# Commit Hash

# What is Commit Hash?

A commit hash is a unique identifier generated by Git for every commit.

Also called:
- SHA hash
- SHA-1 hash
- Commit ID

---

# Characteristics of Commit Hash

| Property | Description |
|---|---|
| Unique | No two commits share same hash |
| Immutable | Changes create new hash |
| Generated Automatically | Git creates hash |
| Long Identifier | Usually 40 hexadecimal characters |

---

# Example Commit Hash

```text
f4c3b00c9e8d3f1a2b4c5d6e7f8g9h0i1j2k3l4
```

Short version:

```text
f4c3b00
```

---

# Purpose of Commit Hash

Hashes are used to:
- Identify commits
- Compare commits
- Restore versions
- Navigate history
- Perform merges

---

# Why Hash Changes?

Even a small change:
- File modification
- Message update
- Metadata change

creates a completely different hash.

---

# Hash Generation

Git uses:
- SHA-1 hashing algorithm

to generate commit IDs.

---

# Benefits of Commit Hash

## 1. Data Integrity

Detects changes automatically.

---

## 2. Unique Identification

Every commit uniquely identifiable.

---

## 3. Security

Hard to duplicate or forge.

---

# HEAD Pointer

# What is HEAD?

HEAD is a special pointer in Git that points to:
- Current branch
or
- Current commit

It represents the current working position in repository history.

---

# Purpose of HEAD

HEAD helps Git know:
- Current branch
- Current commit
- Current working state

---

# HEAD in Normal State

Usually HEAD points to branch:

```text
HEAD → main → Commit C
```

---

# HEAD Movement

HEAD changes when:
- Switching branches
- Checking out commits
- Resetting history

---

# HEAD File

Git stores HEAD information inside:

```text
.git/HEAD
```

---

# Example HEAD Content

```text
ref: refs/heads/main
```

---

# Types of HEAD States

| State | Description |
|---|---|
| Attached HEAD | Points to branch |
| Detached HEAD | Points directly to commit |

---

# Benefits of HEAD Pointer

- Tracks current location
- Simplifies navigation
- Enables branch switching
- Helps commit management

---

# Amending Commits

# What is Amending Commit?

Amending means modifying the most recent commit.

Used for:
- Updating commit message
- Adding forgotten changes
- Correcting mistakes

---

# Purpose of Amend

Amending helps:
- Fix commit messages
- Add missing files
- Improve commit quality

---

# Amend Command

```bash
git commit --amend
```

---

# What Happens During Amend?

Git:
- Replaces previous commit
- Creates new commit hash
- Updates commit history

---

# Common Uses

## 1. Fix Typo in Message

Correct spelling mistakes.

---

## 2. Add Forgotten File

Include missed changes.

---

## 3. Update Small Changes

Modify latest commit quickly.

---

# Important Warning

Avoid amending commits already pushed publicly because:
- Commit hash changes
- Team synchronization problems may occur

---

# Amending Commit Message Only

```bash
git commit --amend -m "New message"
```

---

# Amending with Additional Changes

```bash
git add .
git commit --amend
```

---

# Commit History Navigation

# What is Commit History Navigation?

Commit history navigation means moving through previous commits and repository states.

Git allows developers to:
- View older versions
- Inspect changes
- Restore history
- Analyze development progress

---

# Navigation Methods

Git navigation uses:
- Commit hashes
- HEAD references
- Relative references
- Branch names

---

# Important Navigation References

| Reference | Meaning |
|---|---|
| HEAD | Current commit |
| HEAD~1 | One commit before |
| HEAD~2 | Two commits before |
| branch-name | Branch reference |

---

# Viewing History

## Full History

```bash
git log
```

---

## Short History

```bash
git log --oneline
```

---

## Graph History

```bash
git log --graph
```

---

# Moving Between Commits

```bash
git checkout commit-id
```

---

# Moving Back to Branch

```bash
git checkout main
```

---

# Benefits of Navigation

- Debugging
- Version comparison
- Recovery
- History analysis

---

# Comparison Output

# Benefits of Comparison

# Detached HEAD State

# What is Detached HEAD?

Detached HEAD occurs when HEAD points directly to a commit instead of a branch.

Example:

```text
HEAD → Commit A
```

instead of:

```text
HEAD → main → Commit A
```

---

# How Detached HEAD Happens

Occurs when:
- Checking out specific commit
- Viewing older history
- Temporary navigation

---

# Example Command

```bash
git checkout commit-id
```

---

# Detached HEAD Behavior

In detached HEAD:
- Changes can still be made
- New commits can be created
- But commits are not attached to branch automatically

---

# Risks of Detached HEAD

Commits may become unreachable if:
- User switches branches
- No branch references commit

---

# Example Detached Workflow

```text
main → Commit C

Checkout Commit A

HEAD → Commit A
```

---

# Returning from Detached HEAD

```bash
git checkout main
```

---

# Saving Detached HEAD Work

Create branch before leaving:

```bash
git checkout -b new-branch
```

---

# Benefits of Detached HEAD

Useful for:
- Testing old versions
- Debugging
- Temporary experiments
- Historical inspection

---

# Difference Between Attached and Detached HEAD

| State | HEAD Points To |
|---|---|
| Attached HEAD | Branch |
| Detached HEAD | Commit |
