# 7. Branching in Git

# What is a Branch

# Definition

A branch in Git is an independent line of development.

Branches allow developers to:
- Work on features separately
- Fix bugs independently
- Experiment safely
- Collaborate without affecting main code

---

# Simple Meaning

A branch is like:
- A parallel version of the project

Every branch has:
- Its own commits
- Its own history progression

---

# Default Branch

When a repository is created, Git creates a default branch.

Common default branch names:
- main
- master

---

# Branch Structure

Example:

```text
main
  ├── Commit A
  ├── Commit B
  └── Commit C
```

Create feature branch:

```text
main
  ├── Commit A
  ├── Commit B
  └── Commit C
           \
            feature-login
                ├── Commit D
                └── Commit E
```

---

# How Branches Work Internally

A branch is simply:
- A movable pointer to commits

When new commits are added:
- Branch pointer moves forward

---

# Benefits of Branches

## 1. Isolated Development

Features developed independently.

---

## 2. Safe Experimentation

Test ideas without affecting stable code.

---

## 3. Better Collaboration

Multiple developers work simultaneously.

---

## 4. Organized Workflow

Different tasks separated cleanly.

---

# Importance of Branching

# Why Branching is Important

Branching is one of Git’s most powerful features.

It allows:
- Parallel development
- Safe collaboration
- Feature isolation
- Controlled releases

---

# Problems Without Branching

Without branches:
- All developers edit same code directly
- High conflict risk
- Unstable production code
- Difficult collaboration

---

# Advantages of Branching

| Advantage | Description |
|---|---|
| Isolation | Separate development |
| Stability | Main branch stays safe |
| Collaboration | Multiple developers work together |
| Flexibility | Easy experimentation |
| Faster Development | Parallel workflows |

---

# Real World Example

Suppose team works on:
- Login feature
- Payment gateway
- Dashboard redesign

Each feature gets separate branch.

This prevents:
- Interference
- Code conflicts
- Production instability

---

# Creating Branches

# What is Creating Branch?

Creating branch means making a new development line from current commit.

New branch initially:
- Points to same commit as current branch

---

# Create Branch

```bash
git branch branch-name
```

Example:

```bash
git branch feature-login
```

---

# Create and Switch Branch

```bash
git checkout -b branch-name
```

Example:

```bash
git checkout -b feature-payment
```

---

# Modern Alternative

```bash
git switch -c branch-name
```

---

# Branch Creation Workflow

```text
Current Branch
        ↓
Create New Branch
        ↓
New Branch Points to Same Commit
```

---

# Important Note

Creating branch does NOT:
- Automatically switch branch

unless using:
- `checkout -b`
- `switch -c`

---

---

# Switching Branches

# What is Switching Branches?

Switching branches changes current working branch.

Git updates:
- Working directory
- HEAD pointer
- Current files

---

# Switch Branch

```bash
git checkout branch-name
```

Example:

```bash
git checkout feature-login
```

---

# Modern Switch Command

```bash
git switch branch-name
```

---

# Switch to Previous Branch

```bash
git checkout -
```

---

# What Happens During Switch?

Git:
- Changes HEAD
- Loads branch files
- Updates working directory

---

# Important Notes

Switching may fail if:
- Uncommitted conflicting changes exist

---

# Renaming Branches

# What is Renaming Branch?

Changing branch name without losing history.

---

# Rename Current Branch

```bash
git branch -m new-name
```

---

# Rename Different Branch

```bash
git branch -m old-name new-name
```

---

# Rename Remote Branch Workflow

1. Rename locally
2. Push new branch
3. Delete old remote branch

---

# Push Renamed Branch

```bash
git push origin new-name
```

---

# Delete Old Remote Branch

```bash
git push origin --delete old-name
```

---

# Benefits of Renaming

- Better readability
- Consistent naming
- Cleaner workflows

---

# Deleting Branches

# What is Deleting Branch?

Removing unnecessary branches from repository.

---

# Delete Local Branch

```bash
git branch -d branch-name
```

---

# Force Delete Branch

```bash
git branch -D branch-name
```

---

# Delete Remote Branch

```bash
git push origin --delete branch-name
```

---

# Why Delete Branches?

Used after:
- Feature merge
- Bug fix completion
- Release completion

---

# Benefits of Cleanup

- Cleaner repository
- Easier navigation
- Better management

---

# Branch Workflow

# What is Branch Workflow?

Branch workflow defines:
- How branches are created
- How development happens
- How changes are merged

---

# Basic Workflow

```text
main
   ↓
Create Feature Branch
   ↓
Develop Feature
   ↓
Commit Changes
   ↓
Merge into main
   ↓
Delete Feature Branch
```

---

# Typical Team Workflow

```text
Developer Creates Branch
        ↓
Implements Feature
        ↓
Pushes Branch
        ↓
Creates Pull Request
        ↓
Code Review
        ↓
Merge into Main
```

---

# Workflow Benefits

- Safer development
- Better reviews
- Cleaner releases
- Organized teamwork

---

# Branch Best Practices

# 1. Keep Main Branch Stable

Main branch should always remain production-ready.

---

# 2. Use Meaningful Names

Good examples:

```text
feature-login
bugfix-navbar
hotfix-payment-error
```

---

# 3. Create Small Focused Branches

One branch per feature/task.

---

# 4. Delete Merged Branches

Avoid repository clutter.

---

# 5. Pull Latest Changes Frequently

Reduce merge conflicts.

---

# 6. Avoid Long-Lived Branches

Long branches increase conflict risk.

---

# 7. Commit Frequently

Smaller commits improve tracking.

---
